# Domain Entities and Relationships

This document describes the core domain entities under `server/Express/domain/entities/` and how they relate to one another. Field shapes are informed by both the TypeScript entity classes and the database schema defined in `server/Express/infrastructure/migrations/initializeSchema.ts`.

## Entities

- **User** (`domain/entities/User.ts`)
  - Fields
    - `id: string` – primary identifier
    - `name: string`
    - `password: string` – hashed (not exposed via `toJSON()`)
  - Notes
    - Factory `fromDbRow()` supports optional `password`.

- **Room** (`domain/entities/Room.ts`)
  - Fields
    - `id: string` – UUID (generated by domain if not provided)
    - `name: string`
    - `creatorId: string` – FK to `users.id`
    - `createdAt: number` – epoch ms
    - `users: User[]` – in-memory convenience list for membership
    - `type: 'room' | 'user'` – default `'room'`
    - `isPublic: boolean` – default derived from type (public for `'room'`)
  - Notes
    - DB schema table `rooms` has `type` (TEXT/VARCHAR) and `isPublic` (INTEGER/TINYINT) with defaults.

- **Message** (`domain/entities/Message.ts`)
  - Fields
    - `id: string` – set from DB auto-increment id (stored as string in domain)
    - `author: User`
    - `content: string`
    - `timestamp: number` – epoch ms
    - `status: 'sent' | 'delivered' | 'read'`
    - `sentAt?: number` – defaults to `timestamp`
    - `deliveredAt?: number`
    - `readAt?: number`
  - Notes
    - DB schema `messages` includes: `authorId`, `authorName`, `roomId`, `status`, timestamp columns. Domain reconstructs `author: User` from row (`authorId`, `authorName`).

- **UserSession** (`domain/entities/UserSession.ts`)
  - Fields
    - `id: string`
    - `userId: string` – FK to `users.id`
    - `token: string` – unique
    - `createdAt: number`
    - `expiresAt?: number`
    - `refreshToken?: string` – unique (SQLite path)
    - `refreshTokenExpiresAt?: number`
    - `user?: User` – optional hydrated relation

- **Friend** (`domain/entities/Friend.ts`)
  - Fields
    - `id: string`
    - `userA: string` – FK to `users.id`
    - `userB: string` – FK to `users.id`
    - `status: 'pending' | 'accepted'`
    - `requesterId: string` – the user who initiated the friendship
    - `createdAt: number`
    - `updatedAt: number`
  - Notes
    - Unique composite `(userA, userB)` prevents duplicate edges.
    - Helper `otherUserId(forUserId)` returns the counterpart.

- (Join) **UserRoom** (implicit, table `user_rooms`)
  - Fields
    - `userId: string` – FK to `users.id`
    - `roomId: string` – FK to `rooms.id`
    - Composite PK `(userId, roomId)`
  - Notes
    - Implements many-to-many membership between users and rooms.

## Relationships (Conceptual)

- **User → Room (creator)**
  - One `User` can create many `Room` records via `rooms.creatorId`.
- **User ↔ Room (membership)**
  - Many-to-many via `user_rooms`.
- **Room → Message**
  - One `Room` contains many `Message` records (`messages.roomId`).
- **User → Message (author)**
  - A `Message` is authored by a `User`. Persisted as (`authorId`, `authorName`); mapped to domain `author: User` at read time.
- **User → UserSession**
  - One `User` has many `UserSession` entries (active logins/devices), with unique `token` (and `refreshToken` on SQLite).
- **User ↔ User (Friend)**
  - Self-referential relation through `friends(userA, userB)`. Each edge has `status` and `requesterId`.

## Cardinalities

- `User (1) ── creates ──> (N) Room`
- `User (M) ── member of ──< (N) Room` via `user_rooms`
- `Room (1) ── contains ──> (N) Message`
- `User (1) ── authors ──> (N) Message`
- `User (1) ── has ──> (N) UserSession`
- `User (N) ── friends ── (N) User` through `Friend`

## Indexing summary (from schema)

- Messages: `(roomId, timestamp)` for pagination and `authorId`
- Rooms: `creatorId`, `isPublic`, `name`
- UserRooms: `roomId`
- UserSessions: `userId`, unique `token` (and `refreshToken` on SQLite)
- Friends: `userA`, `userB`, and composite `(userA, status)`, `(userB, status)`

## Notes on domain vs persistence

- The domain models use rich types (`author: User`, `users: User[]`), while persistence normalizes with FKs and primitive columns. Mappers in `domain/dto/mappers/*` adapt entities to API DTOs; repositories adapt DB rows to entities.
